# Формализованные требования — Банковское приложение

## 1. Обзор системы

Система представляет собой банковское приложение, построенное на микросервисной архитектуре. Включает серверную часть (Java) и мобильный клиент (iOS/Swift).

### 1.1. Сервисы

| # | Сервис | Назначение | UI | Своя БД |
|---|--------|-----------|-----|---------|
| 1 | **core-service** | Ядро: счета, балансы, история операций | Нет (только API) | Да (PostgreSQL) |
| 2 | **auth-service** | SSO-аутентификация (OAuth 2.0 Authorization Server) | Да (страница логина) | Да (PostgreSQL) |
| 3 | **user-service** | Данные пользователей (клиенты, сотрудники) | Нет (только API) | Да (PostgreSQL) |
| 4 | **credit-service** | Кредиты, тарифы, кредитный рейтинг | Нет (только API) | Да (PostgreSQL) |
| 5 | **client-bff** | BFF для мобильного приложения клиента банка | Нет (API для iOS) | Да (настройки UI) |
| 6 | **employee-bff** | BFF для мобильного приложения сотрудника банка | Нет (API для iOS) | Да (настройки UI) |

### 1.2. Инфраструктура

| Компонент | Назначение |
|-----------|-----------|
| **PostgreSQL 17** | Реляционная СУБД. Каждый сервис имеет свою БД |
| **Redis** | Кеширование, хранение сессий (в BFF-сервисах) |
| **Apache Kafka** | Брокер сообщений для операций по счетам |
| **Docker Compose** | Оркестрация инфраструктуры для локальной разработки |

## 2. Архитектурные требования

### AR-1. Изоляция данных
Каждый сервис работает только со своей базой данных. Доступ к данным другого сервиса — исключительно через его REST API.

### AR-2. Сетевое взаимодействие
Все межсервисное взаимодействие должно происходить по сети (HTTP REST, WebSocket, Kafka). Прямые вызовы между сервисами в рамках одного процесса запрещены.

### AR-3. Event-driven операции
Добавление новых операций по счёту проходит через Kafka. Последовательность: запись в Kafka → чтение core-service → обработка → сохранение в БД. Гарантия: ни одна операция не должна быть потеряна.

### AR-4. Мастер-счёт банка
Мастер-счёт — банковский счёт, принадлежащий банку. При выдаче кредита деньги переводятся с мастер-счёта на счёт клиента. На мастер-счёте также не допускается отрицательный баланс.

### AR-5. BFF-паттерн
client-bff и employee-bff — серверная часть клиентских приложений. Хранят пользовательские настройки (тема, скрытые счета). Проксируют запросы к core/credit/user сервисам. При установке приложения на новом устройстве настройки загружаются с BFF.

## 3. Функциональные требования

### 3.1. Роль: Клиент

| ID | Требование | Описание |
|----|-----------|----------|
| **FR-C-01** | Открытие счёта | Клиент может открыть один или несколько счетов в банке |
| **FR-C-02** | Закрытие счёта | Клиент может закрыть свой счёт |
| **FR-C-03** | Взятие кредита | Клиент может взять кредит по одному из доступных тарифов |
| **FR-C-04** | Погашение кредита | Клиент может досрочно погасить кредит |
| **FR-C-05** | Внесение средств | Клиент может внести деньги на свой счёт (упрощённо — кнопка) |
| **FR-C-06** | Снятие средств | Клиент может снять деньги со своего счёта (упрощённо — кнопка) |
| **FR-C-07** | История операций | Клиент может просмотреть историю операций по счёту |
| **FR-C-08** | Перевод средств | Клиент может перевести деньги между своими счетами и на чужой счёт |
| **FR-C-09** | Конвертация валют | Клиент может перевести деньги со счёта в одной валюте на счёт в другой валюте с конвертацией по текущему курсу |
| **FR-C-10** | Просмотр кредитов | Клиент может видеть свои кредиты, просроченные платежи и кредитный рейтинг |
| **FR-C-11** | Настройки UI | Клиент может выбрать тему (тёмная/светлая) и скрыть определённые счета |

### 3.2. Роль: Сотрудник

| ID | Требование | Описание |
|----|-----------|----------|
| **FR-E-01** | Просмотр всех счетов | Сотрудник может просмотреть все счета всех клиентов |
| **FR-E-02** | История операций | Сотрудник может просмотреть историю операций по любому счёту |
| **FR-E-03** | Управление тарифами | Сотрудник может создать кредитный тариф (название, процентная ставка) |
| **FR-E-04** | Управление пользователями | Сотрудник может создавать и блокировать клиентов и сотрудников |
| **FR-E-05** | Просмотр кредитов | Сотрудник может просмотреть кредиты любого клиента с детализацией и просроченными платежами |
| **FR-E-06** | Кредитный рейтинг | Сотрудник может просмотреть кредитный рейтинг любого клиента |
| **FR-E-07** | Настройки UI | Сотрудник может выбрать тему и скрыть счета |

## 4. Нефункциональные требования

### 4.1. Аутентификация и авторизация

| ID | Требование |
|----|-----------|
| **NFR-AUTH-01** | SSO: пользователь использует один аккаунт для всех приложений. Один человек может быть одновременно клиентом и сотрудником |
| **NFR-AUTH-02** | Логин/пароль вводится ТОЛЬКО на странице auth-service. Ни один другой сервис не имеет доступа к паролю. Мобильные приложения перенаправляют на auth-service для входа |
| **NFR-AUTH-03** | Протокол: OAuth 2.0 Authorization Code Flow (с PKCE для мобильных клиентов) |
| **NFR-AUTH-04** | Авторизация в каждом сервисе: каждый запрос проверяется на наличие прав (включая межсервисные запросы) |

### 4.2. Real-time

| ID | Требование |
|----|-----------|
| **NFR-RT-01** | Список операций по счёту доставляется клиенту через WebSocket |
| **NFR-RT-02** | При создании новой операции она мгновенно отображается в приложении (push update) |

### 4.3. Целостность данных

| ID | Требование |
|----|-----------|
| **NFR-DATA-01** | Баланс счёта не может быть отрицательным ни при каких обстоятельствах |
| **NFR-DATA-02** | Ни одна операция не должна быть потеряна (Kafka гарантия) |
| **NFR-DATA-03** | Баланс мастер-счёта не может быть отрицательным |

### 4.4. Мультивалютность

| ID | Требование |
|----|-----------|
| **NFR-CURR-01** | Поддержка минимум 3 валют |
| **NFR-CURR-02** | Конвертация по текущему курсу из открытого API |

### 4.5. Кредиты

| ID | Требование |
|----|-----------|
| **NFR-CREDIT-01** | Автоматическое списание оплаты кредита раз в день (или в минуту — для тестирования) |
| **NFR-CREDIT-02** | Кредитный рейтинг — функция, показывающая вероятность возврата кредита (формула на усмотрение разработчика) |
| **NFR-CREDIT-03** | Деньги по кредиту выдаются с мастер-счёта банка |

## 5. Требования к клиентскому приложению (iOS)

### 5.1. Архитектура (слоёная)

```
┌─────────────────────────────────┐
│     UI Layer (SwiftUI Views)    │  ← Переиспользуемые компоненты
├─────────────────────────────────┤
│  Business Logic (Use Cases)     │  ← Сценарии использования
├─────────────────────────────────┤
│  Server Interaction Layer       │
│  ┌────────────────────────────┐ │
│  │ Business Logic of Requests │ │  ← Логика отправки/обработки
│  ├────────────────────────────┤ │
│  │ Network Layer (URLSession) │ │  ← Инкапсуляция сетевых запросов
│  └────────────────────────────┘ │
└─────────────────────────────────┘
```

| ID | Требование |
|----|-----------|
| **NFR-IOS-01** | Строгое разделение слоёв: UI не делает сетевых запросов напрямую |
| **NFR-IOS-02** | Переиспользуемые UI-компоненты (модалки, формы, таблицы, подтверждения) — не копипаст |
| **NFR-IOS-03** | Типобезопасные DTO: использование `any` запрещено. Строгая типизация запросов/ответов |
| **NFR-IOS-04** | Единый механизм обработки ошибок: иерархия/конвейер (компонент → страница → глобальный обработчик → fallback) |
| **NFR-IOS-05** | Каждое действие пользователя имеет явную реакцию: loading, success или error. Нет «тихих» ошибок |

## 6. Распределение функционала по сервисам

### core-service
- CRUD счетов (открытие, закрытие)
- Балансы всех счетов
- Пополнение / снятие со счёта
- Переводы между счетами (включая мультивалютные)
- История операций (запись через Kafka consumer)
- Мастер-счёт банка
- WebSocket endpoint для push операций

### auth-service
- OAuth 2.0 Authorization Server
- Страница логина (Thymeleaf)
- Управление токенами (access, refresh)
- Регистрация (при создании пользователя через user-service)

### user-service
- CRUD пользователей (клиенты, сотрудники)
- Блокировка/разблокировка пользователей
- Роли и привилегии
- Профиль пользователя

### credit-service
- CRUD кредитных тарифов
- Выдача кредитов (списание с мастер-счёта через core-service API)
- Автоматическое списание ежедневных платежей (scheduler)
- Отслеживание просроченных платежей
- Кредитный рейтинг

### client-bff
- Проксирование запросов клиента к core/credit/user сервисам
- Хранение пользовательских настроек (тема, скрытые счета)
- WebSocket проксирование для real-time операций
- Авторизация: доступ только к своим данным

### employee-bff
- Проксирование запросов сотрудника к core/credit/user сервисам
- Хранение пользовательских настроек сотрудника
- WebSocket проксирование
- Авторизация: доступ ко всем данным клиентов
