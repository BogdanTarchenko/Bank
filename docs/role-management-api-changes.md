# Изменения в API: управление ролями и контроль доступа

## Краткое описание

Внесены три изменения в поведение бэкенда, которые **требуют обновления клиентских приложений**:

1. [Новый endpoint](#1-новый-endpoint-управления-ролями) — управление ролями пользователей через employee-bff
2. [Инвалидация токенов](#2-инвалидация-токенов-после-изменения-ролей) — после смены ролей токен становится недействительным
3. [Требование роли CLIENT](#3-требование-роли-client-для-client-bff) — client-bff теперь требует роль CLIENT

---

## 1. Новый endpoint управления ролями

### Для: Employee Frontend (employee-bff, порт 8085)

Добавлен выделенный endpoint для изменения ролей пользователя. Прямой проксированный вызов через `/api/v1/proxy/user/users/{id}/roles` **заблокирован** и вернёт `403 Forbidden`.

### Endpoint

```
PATCH /api/v1/users/{id}/roles
Host: employee-bff:8085
Authorization: Bearer <token>
Content-Type: application/json
```

### Тело запроса

```json
{
  "roles": ["EMPLOYEE", "CLIENT"]
}
```

Допустимые значения ролей: `"CLIENT"`, `"EMPLOYEE"`.
Поле `roles` обязательно и не может быть пустым.

### Примеры

**Назначить пользователю только роль CLIENT:**
```json
{ "roles": ["CLIENT"] }
```

**Назначить пользователю роли CLIENT и EMPLOYEE:**
```json
{ "roles": ["CLIENT", "EMPLOYEE"] }
```

### Ответы

| Статус | Описание |
|--------|----------|
| `200 OK` | Роли успешно обновлены. Тело — объект `UserResponse` |
| `400 Bad Request` | Некорректный формат запроса или пустой список ролей |
| `403 Forbidden` | Сотрудник пытается снять с себя роль EMPLOYEE (см. ниже) |
| `404 Not Found` | Пользователь с указанным ID не найден |
| `401 Unauthorized` | Токен отсутствует или недействителен |

### Ограничение: запрет самодеградации

Сотрудник **не может** убрать роль `EMPLOYEE` у самого себя.
При попытке — ответ `403 Forbidden`:

```json
{ "error": "Сотрудник не может снять с себя роль EMPLOYEE" }
```

Убрать роль `EMPLOYEE` у сотрудника может **только другой сотрудник**.

### Изменение маршрутизации в employee-bff

| Было | Стало |
|------|-------|
| `PATCH /api/v1/proxy/user/users/{id}/roles` (проксировалось, но endpoint в user-service не существовал) | `PATCH /api/v1/users/{id}/roles` (прямой endpoint на employee-bff) |

Все остальные `PATCH` на `/api/v1/proxy/user/users/{id}/*` (например `/block`, `/unblock`) **не затронуты**.

## 2. Инвалидация токенов после изменения ролей

### Для: Employee Frontend и Client Frontend (оба BFF)

После того как сотрудник изменяет роли пользователя через `PATCH /api/v1/users/{id}/roles`, все токены этого пользователя, выданные **до момента изменения**, становятся недействительными.

### Поведение

При следующем запросе с устаревшим токеном оба BFF вернут:

```
HTTP 401 Unauthorized
Content-Type: application/json

{ "error": "Токен недействителен. Выполните повторный вход." }
```

### Что нужно сделать на стороне фронтенда

При получении `401` с текстом `"Токен недействителен"`:

1. Попытаться обновить токен через refresh flow (`/oauth2/token` с `grant_type=refresh_token`).
2. Если refresh тоже вернёт ошибку — перенаправить пользователя на страницу входа.
3. **Не повторять** оригинальный запрос бесконечно — обрабатывать это как сессионную ошибку.

> **Важно:** обычный `401` (токен просрочен) и `401` с телом `"Токен недействителен"` — семантически одинаковы с точки зрения клиента. Оба требуют повторной аутентификации.

### Сценарий

```
1. Сотрудник A (токен выдан в 10:00) работает в системе
2. Сотрудник B в 10:30 изменяет роли сотрудника A (убирает EMPLOYEE)
3. Сотрудник A отправляет запрос в 10:31 со своим токеном
4. employee-bff возвращает 401 — сессия инвалидирована
5. Клиент сотрудника A должен сделать повторный вход
6. После входа новый токен содержит актуальные роли (без EMPLOYEE)
7. employee-bff вернёт 403 на все запросы (нет роли EMPLOYEE)
```

## 3. Требование роли CLIENT для client-bff

### Для: Client Frontend (client-bff, порт 8084)

**client-bff теперь требует роль `CLIENT`** для доступа ко всем endpoint'ам (кроме WebSocket `/ws/**`).

### Было / Стало

| | Было | Стало |
|---|------|-------|
| Требование доступа | Любой аутентифицированный пользователь | Только пользователи с ролью `CLIENT` |
| Ответ для пользователя без `CLIENT` | `200 OK` (доступ разрешён) | `403 Forbidden` |

### Затронутые сценарии

- Пользователь с только ролью `EMPLOYEE` (без `CLIENT`) — получит `403` при обращении к client-bff.
- Пользователь без какой-либо роли — получит `403`.
- Пользователь с ролью `CLIENT` (с `EMPLOYEE` или без) — доступ разрешён.

### Что нужно сделать на стороне фронтенда

- Убедиться, что при регистрации клиентам назначается роль `CLIENT` (это поведение по умолчанию — без изменений).
- При получении `403` от client-bff — показать пользователю сообщение об ошибке доступа (не путать с ошибкой аутентификации).
